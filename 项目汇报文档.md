# 地铁路径规划系统项目汇报文档

## 1. 项目概述

本项目是一个基于 C++ 开发的地铁路径规划系统，主要面向地铁线路查询、站点状态维护和路径分析场景。程序通过读取 CSV 数据文件，完成站点信息加载、线路连接关系建图，并基于图算法实现最短时间路径规划、最少换乘路径规划、多条候选路径查询以及站点关闭影响分析等功能。

项目整体采用模块化设计，分为：

- `common.h`：公共常量、数据结构与通用工具函数
- `station.h / station.cpp`：站点信息管理模块
- `graph.h / graph.cpp`：地铁线路图构建模块
- `pathfinder.h / pathfinder.cpp`：路径搜索与影响分析模块
- `menu.h / menu.cpp`：控制台菜单交互模块
- `main.cpp`：程序入口与系统初始化模块

## 2. 项目主要功能

本项目实现了以下核心功能：

1. 从 `Station.csv` 中读取地铁站点信息。
2. 从 `Edge.csv` 中读取站点连接关系并构建地铁图。
3. 支持按 CSV 文件批量更新站点开启/关闭状态。
4. 支持手工修改指定站点的运行状态。
5. 支持查看当前所有关闭站点。
6. 支持将站点状态恢复为初始数据。
7. 支持按线路查看该线路全部站点及换乘线路信息。
8. 支持按站名关键字搜索站点。
9. 支持查询单条最短时间路径。
10. 支持查询 3 条最短时间候选路径。
11. 支持查询单条最少换乘路径。
12. 支持查询 3 条最少换乘候选路径。
13. 支持分析某一关闭站点对周边站点的直接影响。

## 3. 系统设计思路

### 3.1 总体设计思路

本系统采用“数据管理 + 图建模 + 路径搜索 + 菜单交互”的分层设计思路。

- 首先由 `StationManager` 读取站点数据，维护所有站点的编号、名称、线路和状态。
- 然后由 `MetroGraph` 读取边数据，将地铁网络抽象为有向图的邻接表结构。
- 再由 `PathFinder` 基于图结构进行路径规划与站点影响分析。
- 最后由 `MenuSystem` 提供控制台交互入口，将用户输入映射到具体业务功能。

这样的设计具有以下优点：

- 模块职责清晰，便于分工和维护。
- 数据层、算法层和交互层相互独立，耦合度较低。
- 后续如果要扩展 GUI、Web 前端或数据库存储，可在现有结构上继续演进。

### 3.2 数据结构设计思路

系统中最重要的几个数据结构如下：

- `Station`：表示单个站点，包含站点编号、名称、所属线路和状态。
- `Edge`：表示站点间连接关系，包含起点、终点、线路名、方向、时间代价和边类型。
- `PathResult`：封装路径查询结果，包括站点序列、站名序列、线路序列、换乘站、总时间、换乘次数、经过站数和路径是否有效。

设计这些结构的目的，是把“原始 CSV 数据”转换成“可供算法直接使用的对象模型”，便于后续查询和展示。

### 3.3 图模型设计思路

本项目将地铁网络抽象为一个有向图：

- 每个站点视为一个顶点。
- 相邻站点之间的连接关系视为一条边。
- 换乘关系也用边表示，并通过 `EdgeType::Transfer` 与普通线路边区分。
- 使用邻接表 `vector<vector<Edge>>` 存储图，既节省空间，也便于遍历相邻站点。

图模型的优势在于：

- 能够很好表达复杂线路连接关系。
- 便于使用 Dijkstra、优先队列搜索等经典图算法。
- 在站点数较多时，邻接表结构比邻接矩阵更高效。

### 3.4 路径规划设计思路

项目中实现了两类核心路径规划目标：

1. 最短时间路径
2. 最少换乘路径

对于单条最优路径查询，系统使用“带优先级的最短路搜索思想”：

- 最短时间路径：以总时间为第一优化目标，以换乘次数为第二优化目标。
- 最少换乘路径：以换乘次数为第一优化目标，以总时间为第二优化目标。

对于 3 条候选路径查询，系统通过优先队列保存候选状态，不断扩展路径，并限制重复扩展次数，从而找到多条较优但不完全相同的路径。

### 3.5 站点状态管理设计思路

现实中的地铁系统会出现站点临时关闭、施工维护、运营调整等情况，因此本系统引入了站点状态字段：

- `Open`：开启
- `Closed`：关闭

所有路径规划都会先检查站点是否可用。如果起点、终点或中间站点关闭，则该站点不会参与路径搜索。这使系统更贴近真实业务场景。

## 4. 项目模块说明

### 4.1 公共模块 `common.h`

该模块主要负责定义系统级常量、枚举类型、数据结构以及通用工具函数。

- `INF`：表示一个极大值，用于最短路初始化。
- `TRANSFER_TIME`：定义换乘时间常量，当前版本中未直接参与路径时间计算，保留用于后续扩展。
- `StationStatus`：站点状态枚举，区分开启与关闭。
- `EdgeType`：边类型枚举，区分普通边与换乘边。
- `Station`：站点结构体。
- `Edge`：边结构体。
- `PathResult`：路径查询结果结构体。

### 4.2 站点管理模块 `StationManager`

该模块负责站点数据读取、索引维护、状态修改、线路查询和名称检索。

核心思想是：

- 使用 `vector<Station>` 存储全部站点信息。
- 使用 `unordered_map<int, int>` 建立“站点 id -> 数组下标”的索引，加快查询速度。

### 4.3 图管理模块 `MetroGraph`

该模块负责将 CSV 中的边关系转换为程序中的图结构。  
它的核心是邻接表 `adj_`，其中 `adj_[i]` 表示编号为 `i` 的站点可到达的所有边。

### 4.4 路径搜索模块 `PathFinder`

该模块是本项目的算法核心，负责：

- 最短时间路径查询
- 最少换乘路径查询
- 多条候选路径查询
- 关闭站点影响分析

### 4.5 菜单交互模块 `MenuSystem`

该模块负责和用户交互，包括主菜单、子菜单显示、输入校验、结果输出和暂停等待等功能。

### 5.4 `graph.cpp / graph.h`

#### `bool MetroGraph::buildFromCsv(const std::string& fileName, const StationManager& stationManager)`

功能：

- 从边文件中读取站点连接关系并构建邻接表图。

主要流程：

- 打开边数据文件。
- 根据站点最大 id 初始化邻接表容量。
- 逐行读取边信息：起点、终点、线路、方向、时间。

- 构造 `Edge` 对象并加入邻接表。

设计思路：

- 采用邻接表存图，适合稀疏图搜索。
- 建图时先校验站点合法性，防止脏数据进入系统。

#### `PathResult PathFinder::shortestTimePath(int startId, int endId) const`

功能：

- 查询单条总耗时最短路径。

算法思路：

- 使用优先队列进行最短路搜索。
- 以总时间为主关键字。
- 当总时间相同时，以换乘次数更少为优先。

特点：

- 能在满足最短时间目标的同时给出更合理的换乘方案。

#### `PathResult PathFinder::minTransferPath(int startId, int endId) const`

功能：

- 查询单条换乘次数最少路径。

算法思路：

- 仍使用优先队列搜索。
- 以换乘次数为主关键字。
- 当换乘次数相同时，以总时间更少为优先。

#### `std::vector<PathResult> PathFinder::kShortestTimePaths(int startId, int endId, int k) const`

功能：

- 查询 `k` 条总耗时较短的候选路径，默认取 3 条。

算法思路：

- 优先队列中保存当前候选路径状态。
- 每次扩展当前最优候选。
- 通过 `containsStation` 避免重复经过站点。
- 通过 `makePathKey` 防止重复路径加入结果。
- 通过 `expandedCount` 和 `maxExpansions` 控制搜索规模。

说明：  
这是对单最优路径的扩展，适合展示多个备选方案。

#### `std::vector<PathResult> PathFinder::kMinTransferPaths(int startId, int endId, int k) const`

功能：

- 查询 `k` 条换乘次数较少的候选路径，默认取 3 条。

算法思路：

- 与 `kShortestTimePaths` 类似。
- 只是优先级由“时间优先”改为“换乘优先”。

#### `std::vector<int> PathFinder::analyzeAffectedArea(const std::vector<int>& closedStationIds) const`

功能：

- 分析关闭站点集合对周边站点造成的直接影响。

实现思路：

- 遍历所有关闭站点的邻接边。
- 将与关闭站点直接相连、但自身不在关闭集合中的站点加入影响列表。
- 使用集合去重，并按编号排序输出。

## 6. 程序运行流程

项目整体运行流程如下：

1. 程序启动。
2. 加载站点数据文件 `Station.csv`。
3. 加载边数据文件 `Edge.csv` 并构建地铁图。
4. 创建路径规划器。
5. 进入菜单循环。
6. 用户根据菜单选择功能。
7. 对应模块执行查询、修改或分析操作。
8. 输出结果并等待下一次操作。

## 7. 项目亮点

### 7.1 模块划分清晰

项目按“数据、图、算法、交互”分层，结构清楚，便于团队协作开发。

### 7.2 兼顾静态数据与动态运营状态

不仅支持基础路径规划，还支持站点开启/关闭状态变更，更接近真实地铁调度场景。

### 7.3 同时支持两种优化目标

用户可以按“时间最短”或“换乘最少”两种不同需求进行查询，提高系统实用性。

### 7.4 支持多候选路径输出

相比只输出单一路径，系统还能提供 3 条备选路径，提升结果灵活性。

### 7.5 支持影响分析

站点关闭后，系统可以分析直接受影响站点，为运营调整和应急预案提供参考。

## 8. 项目存在的不足

虽然系统已具备完整基本功能，但仍有一些可以继续优化的地方：

1. 当前主要为控制台程序，界面交互较为基础。
2. `TRANSFER_TIME` 常量暂未纳入总时间计算，可进一步完善换乘时间模型。
3. 多条候选路径搜索采用启发式扩展方式，结果较实用，但不是严格意义上的经典 K 最短路径算法。
4. 影响分析目前只分析直接相邻站点，尚未扩展到更大范围的连通性影响。
5. 输入异常处理还可以进一步细化，例如手工更新状态时对非法数字输入的二次校验。

## 9. 后续优化方向

后续可从以下方向继续扩展：

1. 增加图形界面，提高可视化程度。
2. 加入站点间距离、票价等更多维度的路径规划。
3. 将换乘时间纳入总耗时模型，使结果更真实。
4. 增加“避开指定线路/站点”的高级查询功能。
5. 将影响分析扩展为多层传播分析和连通性分析。
6. 接入数据库或在线数据源，提升系统可维护性。

## 10. 总结

本项目完成了一个较完整的地铁路径规划系统实现。系统以 CSV 数据为基础，以图结构为核心，以路径搜索算法为重点，较好地实现了站点状态管理、路径规划、线路查询和影响分析等功能。

从课程设计或项目汇报角度来看，本项目具备以下价值：

- 体现了面向对象设计思想；
- 体现了图结构建模能力；
- 体现了最短路径与多目标优化算法应用；
- 体现了基础业务系统的模块化实现能力。

整体而言，这是一个功能完整、结构清晰、具有较强实践意义的 C++ 项目。
